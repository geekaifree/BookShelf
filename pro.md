## 代码质量

### 整洁代码原则

| 原则 | 描述 | 示例 |
|------|------|------|
| 单一职责 | 一个函数，一个用途 | `calculateTotal()` 而非 `processOrder()` |
| 有意义的命名 | 描述性变量/函数名 | `userEmail` 而非 `e` 或 `data` |
| 小函数 | 函数不超过 20 行 | 将复杂逻辑拆分为辅助函数 |
| DRY | 不要重复自己 | 提取公共代码 |
| KISS | 保持简单 | 避免聪明的一行代码 |
| YAGNI | 你不会需要它 | 不要构建不需要的东西 |

### 代码审查清单

| 检查项 | 描述 |
|--------|------|
| 正确性 | 代码是否做了它应该做的事？ |
| 错误处理 | 错误是否被捕获和处理？ |
| 边界情况 | 边界条件是否处理？ |
| 性能 | 是否有明显的低效？ |
| 安全性 | 是否有漏洞？ |
| 可读性 | 其他人能理解代码吗？ |
| 测试 | 是否有足够的测试？ |
| 文档 | 复杂部分是否解释清楚？ |

### 重构模式

| 模式 | 描述 | 何时使用 |
|------|------|----------|
| 提取函数 | 将代码提取到新函数 | 函数太长 |
| 重命名 | 给更好的名字 | 目的不明确 |
| 内联 | 移除不必要的抽象 | 不必要的间接 |
| 移动 | 将代码移到正确的模块 | 位置错误 |
| 替换为策略 | 使用多态 | 复杂条件判断 |
| 引入参数对象 | 分组相关参数 | 参数太多 |

## 测试

### 测试金字塔

| 层级 | 速度 | 范围 | 数量 |
|------|------|------|------|
| 单元测试 | 快 | 单个函数/类 | 多 |
| 集成测试 | 中 | 组件交互 | 一些 |
| 端到端测试 | 慢 | 完整系统流程 | 少 |

### 测试命名规范

```python
# 模式：test_<单元>_<条件>_<预期结果>
def test_calculate_total_with_empty_cart_returns_zero():
    assert calculate_total([]) == 0

def test_calculate_total_with_multiple_items_returns_sum():
    items = [Item(price=10), Item(price=20)]
    assert calculate_total(items) == 30
```

### 测试策略

| 策略 | 描述 | 使用场景 |
|------|------|----------|
| TDD | 先写测试再写代码 | 新功能 |
| BDD | 行为驱动规范 | 面向用户的功能 |
| 基于属性 | 生成随机输入 | 数据处理 |
| 快照 | 将输出与基准比较 | UI 组件 |
| 变异 | 验证测试有效性 | 测试质量 |

### Mock 最佳实践

| 实践 | 描述 |
|------|------|
| Mock 边界 | Mock 外部服务，而非内部逻辑 |
| 验证交互 | 断言 Mock 被正确调用 |
| 使用测试替身 | 查询用 Stub，命令用 Mock |
| 保持 Mock 简单 | 复杂 Mock 表示设计有问题 |

## 架构

### 架构模式

| 模式 | 描述 | 最适用于 |
|------|------|----------|
| 单体 | 单个可部署单元 | 小团队、简单应用 |
| 微服务 | 独立服务 | 大团队、复杂领域 |
| 无服务器 | 基于函数 | 事件驱动工作负载 |
| 事件驱动 | 异步消息传递 | 解耦系统 |
| 分层 | 水平分离 | CRUD 应用 |
| 六边形 | 端口和适配器 | 可测试的业务逻辑 |

### SOLID 原则

| 原则 | 描述 | 优势 |
|------|------|------|
| 单一职责 | 一个变更的理由 | 可维护性 |
| 开放/封闭 | 对扩展开放，对修改封闭 | 可扩展性 |
| 里氏替换 | 子类型可替换 | 可靠性 |
| 接口隔离 | 小而专注的接口 | 灵活性 |
| 依赖反转 | 依赖抽象 | 可测试性 |

### 设计模式

| 类别 | 模式 | 使用场景 |
|------|------|----------|
| 创建型 | Factory、Builder、Singleton | 对象创建 |
| 结构型 | Adapter、Decorator、Facade | 对象组合 |

## 调试

### 调试流程

| 步骤 | 操作 |
|------|------|
| 1. 复现 | 创建最小复现 |
| 2. 隔离 | 缩小问题范围 |
| 3. 诊断 | 识别根本原因 |
| 4. 修复 | 实施解决方案 |
| 5. 验证 | 确认修复有效 |
| 6. 预防 | 添加测试防止回归 |

### 调试工具

| 工具 | 语言 | 用途 |
|------|------|------|
| gdb | C/C++ | 单步调试 |
| pdb | Python | 交互式调试器 |
| Chrome DevTools | JavaScript | 浏览器调试 |
| strace | Linux | 系统调用跟踪 |
| Wireshark | 网络 | 数据包分析 |
| 日志分析 | 任何 | 结构化日志 |

### 常见 Bug 类别

| 类别 | 症状 | 解决方案 |
|------|------|----------|
| 差一错误 | 错误计数、缺失项 | 检查循环边界 |
| 空引用 | NullPointerException | 添加空值检查 |
| 竞态条件 | 间歇性失败 | 添加同步 |
| 内存泄漏 | 内存使用增长 | 修复资源清理 |
| 逻辑错误 | 错误输出 | 审查算法 |

## 版本控制

### Git 最佳实践

| 实践 | 描述 |
|------|------|
| 原子提交 | 每次提交一个逻辑变更 |
| 有意义的消息 | 解释为什么，而非什么 |
| 功能分支 | 在分支中隔离工作 |
| 合并前变基 | 清洁的线性历史 |
| 合并前审查 | 代码审查所有变更 |

### 提交消息格式

```
<类型>(<范围>): <描述>

<正文>

<页脚>
```

| 类型 | 描述 |
|------|------|
| feat | 新功能 |
| fix | Bug 修复 |
| docs | 文档 |
| style | 格式化 |
| refactor | 代码重构 |
| test | 添加测试 |
| chore | 维护 |

### 分支策略

| 策略 | 描述 | 最适用于 |
|------|------|----------|
| Git Flow | develop、feature、release、hotfix | 基于发布的项目 |
| GitHub Flow | main + 功能分支 | 持续部署 |
| 基于主干 | 短命分支，合并到 main | CI/CD 团队 |

## 总结

专业编程不仅仅是编写可运行的代码。它涵盖整洁代码实践、彻底测试、合理架构、有效调试和持续学习。首先专注于基础：编写可读的代码、彻底测试、理解设计模式、学会系统性调试。这些技能会在职业生涯中不断积累。
