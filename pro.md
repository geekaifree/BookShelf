## 整洁代码

### 原则

| 原则 | 描述 |
|-----------|-------------|
| 单一职责 | 每个函数/类只做一件事 |
| 开闭原则 | 对扩展开放，对修改关闭 |
| 里氏替换 | 子类型必须可替换 |
| 接口隔离 | 多个特定接口优于一个通用接口 |
| 依赖反转 | 依赖抽象而非具体实现 |
| DRY | 不要重复自己 |
| KISS | 保持简单 |
| YAGNI | 你不会需要它 |

### 命名规范

| 元素 | 规范 | 示例 |
|---------|------------|---------|
| 变量 | camelCase 或 snake_case | `userName`、`user_name` |
| 函数 | camelCase 或 snake_case | `getUser()`、`get_user()` |
| 类 | PascalCase | `UserService` |
| 常量 | UPPER_SNAKE | `MAX_RETRY_COUNT` |
| 文件 | kebab-case 或 snake_case | `user-service.py` |

### 代码坏味道

| 坏味道 | 描述 | 解决方案 |
|-------|-------------|----------|
| 长函数 | 函数做了太多事 | 提取更小的函数 |
| 深层嵌套 | 多层缩进 | 使用提前返回 |
| 魔法数字 | 未解释的数值 | 使用命名常量 |
| 重复代码 | 多处相同的代码 | 提取共享函数 |
| 上帝类 | 职责过多的类 | 拆分为更小的类 |
| 长参数列表 | 函数参数过多 | 使用对象/结构体 |

### 格式化最佳实践

| 实践 | 描述 |
|----------|-------------|
| 一致缩进 | 统一使用空格或制表符 |
| 行长度 | 保持在 80-120 字符以内 |
| 空行 | 分隔逻辑部分 |
| 分组 | 相关代码放在一起 |
| 注释 | 解释为什么，而非是什么 |

## 测试

### 测试金字塔

| 层级 | 描述 | 速度 | 覆盖率 |
|-------|-------------|-------|----------|
| 单元测试 | 测试单个函数 | 快 | 高 |
| 集成测试 | 测试组件交互 | 中等 | 中等 |
| 端到端测试 | 测试完整工作流 | 慢 | 低 |

### 单元测试

**好的单元测试的特征：**

| 特征 | 描述 |
|----------------|-------------|
| 快速 | 毫秒级执行 |
| 隔离 | 不依赖外部系统 |
| 可重复 | 每次结果相同 |
| 自验证 | 通过或失败，无需手动检查 |
| 及时 | 与代码同步编写 |

**测试框架：**

| 语言 | 框架 |
|----------|-----------|
| Python | pytest、unittest |
| JavaScript | Jest、Mocha、Vitest |
| Java | JUnit、TestNG |
| Go | testing 包 |
| Rust | 内置测试框架 |

### 测试驱动开发（TDD）

1. **红：** 编写失败的测试
2. **绿：** 编写最少代码通过测试
3. **重构：** 改进代码

### Mock 和 Stub

| 技术 | 描述 | 使用场景 |
|-----------|-------------|----------|
| Stub | 返回预定义数据 | 从依赖中隔离 |
| Mock | 验证交互 | 检查方法调用 |
| Fake | 工作实现 | 数据库、邮件 |
| Spy | 记录调用 | 分析、日志 |

### 代码覆盖率

| 指标 | 描述 | 目标 |
|--------|-------------|--------|
| 行覆盖率 | 执行的行 | 80%+ |
| 分支覆盖率 | 走过的分支 | 70%+ |
| 函数覆盖率 | 调用的函数 | 90%+ |
| 变异覆盖率 | 检测到的变异 | 60%+ |

## 调试

### 调试流程

1. **复现** bug 一致地出现
2. **隔离** 问题区域
3. **识别** 根本原因
4. **修复** 问题
5. **验证** 修复有效
6. **防止** 回归的测试

### 调试工具

| 工具 | 描述 | 使用场景 |
|------|-------------|----------|
| 调试器 | 逐步执行代码 | 复杂逻辑 |
| 日志 | 记录事件 | 生产问题 |
| 性能分析器 | 测量性能 | 性能 bug |
| 内存分析器 | 追踪内存使用 | 内存泄漏 |
| 网络检查器 | 监控网络调用 | API 问题 |

### 常见 Bug 类别

| 类别 | 症状 | 工具 |
|----------|----------|-------|
| 逻辑错误 | 错误输出 | 调试器、测试 |
| 内存泄漏 | 内存持续增长 | 性能分析器、valgrind |
| 竞态条件 | 间歇性失败 | 线程消毒器 |
| 空引用 | 崩溃 | 静态分析 |
| 差一错误 | 边界错误 | 单元测试 |

### 日志最佳实践

| 级别 | 何时使用 |
|-------|-------------|
| ERROR | 出了问题 |
| WARN | 可能出问题 |
| INFO | 重要事件 |
| DEBUG | 详细信息 |
| TRACE | 非常详细的信息 |

| 模式 | 描述 | 使用场景 |

| 层 | 描述 | 依赖 |

| 原则 | 违反 | 修复 |

## 性能优化

### 优化流程

1. **测量** 当前性能
2. **分析** 找到瓶颈
3. **优化** 瓶颈
4. **验证** 改进
5. **重复** 根据需要

### 常见瓶颈

| 瓶颈 | 症状 | 解决方案 |
|------------|----------|-----------|
| CPU | CPU 使用率高 | 优化算法 |
| 内存 | 内存使用率高 | 修复泄漏、优化数据 |
| I/O | 磁盘/网络慢 | 异步、缓存 |
| 数据库 | 查询慢 | 索引、查询优化 |
| 网络 | 传输慢 | 压缩、CDN |

### 缓存策略

| 策略 | 描述 | 使用场景 |
|----------|-------------|----------|
| 内存 | 存储在 RAM 中 | 热数据 |
| 分布式 | Redis、Memcached | 共享缓存 |
| CDN | 边缘服务器 | 静态资源 |
| 浏览器 | 客户端 | 用户特定数据 |
| 数据库 | 查询缓存 | 重复查询 |

### 性能分析工具

| 语言 | 工具 | 功能 |
|----------|------|----------|
| Python | cProfile、py-spy | CPU 分析 |
| JavaScript | Node Clinic、Chrome DevTools | CPU、内存 |
| Java | VisualVM、JProfiler | CPU、内存、线程 |
| Go | pprof | CPU、内存 |
| Rust | perf、cargo-flamegraph | CPU 分析 |

## 安全

### 安全原则

| 原则 | 描述 |
|-----------|-------------|
| 最小权限 | 最低必要权限 |
| 纵深防御 | 多层安全 |
| 安全失败 | 失败时安全默认 |
| 职责分离 | 拆分关键操作 |
| 安全设计 | 从开始就内置 |

### 常见漏洞（OWASP Top 10）

| 漏洞 | 描述 | 预防 |
|---------------|-------------|------------|
| 注入 | SQL、NoSQL、OS 注入 | 参数化查询 |
| 认证失效 | 弱认证 | 强密码、MFA |
| 敏感数据泄露 | 数据泄露 | 静态/传输加密 |
| XSS | 跨站脚本 | 输入净化 |
| CSRF | 跨站请求伪造 | CSRF token |
| 不安全的反序列化 | 不可信数据 | 验证输入 |
| 安全配置错误 | 默认配置 | 安全默认 |

### 认证和授权

| 方法 | 描述 | 使用场景 |
|--------|-------------|----------|
| Session | 服务端会话 | 传统 Web |
| JWT | 无状态令牌 | API、微服务 |
| OAuth 2.0 | 委托授权 | 第三方访问 |
| API 密钥 | 简单令牌 | 服务间通信 |

### 输入验证

| 验证 | 描述 |
|------------|-------------|
| 类型检查 | 确保正确数据类型 |
| 长度检查 | 强制大小限制 |
| 格式检查 | 验证模式（邮箱、电话） |
| 范围检查 | 强制最小/最大值 |
| 白名单 | 仅允许已知的好值 |

## 总结

编程进阶需要在技术和软技能上持续提升。关键要点：

1. **编写整洁代码**，可读且可维护
2. **彻底测试**，使用适当的测试覆盖率
3. **系统调试**，使用正确的工具和技术
4. **审查代码**，提高质量并分享知识
5. **编写文档**，为未来的维护者
6. **选择合适的架构**，满足你的需求
7. **优化性能**，基于测量结果
8. **优先考虑安全**，从一开始就
9. **发展职业**，通过持续学习
10. **保持更新**，跟上行业发展

成为专业程序员的旅程是持续的。专注于基础，定期练习，向他人学习。
