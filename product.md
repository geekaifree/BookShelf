## 设计流程

产品设计流程是迭代的，不是线性的。随着你对用户和问题空间的了解加深，你会重新审视之前的阶段。

### 双钻石模型

双钻石是一个广泛使用的框架：

```
    问题                    解决方案
   发现    定义    开发    交付
      /\            /\            /\            /\
     /  \          /  \          /  \          /  \
    /    \        /    \        /    \        /    \
   /      \      /      \      /      \      /      \
  发散 → 收敛 → 发散 → 收敛
```

| 阶段 | 目标 | 活动 |
|------|------|------|
| 发现 | 理解问题 | 用户研究、市场分析 |
| 定义 | 缩小问题范围 | 综合分析、问题陈述 |
| 开发 | 探索解决方案 | 头脑风暴、原型设计 |
| 交付 | 完善并发布 | 测试、实现、迭代 |

### 设计流程阶段

| 阶段 | 描述 | 关键产出 |
|------|------|----------|
| 研究 | 理解用户和上下文 | 研究发现 |
| 分析 | 将研究综合为洞察 | 用户需求、痛点 |
| 构思 | 生成潜在解决方案 | 概念草图 |
| 原型设计 | 构建可测试的表示 | 交互式原型 |
| 测试 | 用真实用户验证 | 可用性发现 |
| 实现 | 与开发者协作 | 设计规格 |
| 发布 | 向用户发布 | 上线产品 |
| 迭代 | 基于数据改进 | 更新的设计 |

## 用户研究

用户研究是优秀产品设计的基础。它帮助你了解用户是谁、他们需要什么以及他们如何行为。

### 研究方法

| 方法 | 类型 | 使用时机 | 产出 |
|------|------|----------|------|
| 用户访谈 | 定性 | 深入理解需求 | 引用、主题 |
| 问卷调查 | 定量 | 大规模验证假设 | 统计数据 |
| 情境调查 | 定性 | 理解真实环境 | 观察笔记 |
| 卡片分类 | 混合 | 信息架构 | 分类结构 |
| 日记研究 | 定性 | 长期行为模式 | 时间洞察 |
| A/B 测试 | 定量 | 比较设计变体 | 性能指标 |
| 数据分析 | 定量 | 理解当前行为 | 使用模式 |
| 竞品分析 | 混合 | 了解市场 | 功能对比 |

### 用户访谈实施

**准备：**

1. 定义你的研究问题
2. 招募有代表性的参与者（每个用户群 5-8 人）
3. 创建访谈指南（不是脚本）
4. 准备录音设备

**访谈结构：**

| 环节 | 时长 | 目的 |
|------|------|------|
| 开场 | 5 分钟 | 建立信任，说明目的 |
| 背景 | 10 分钟 | 了解上下文和经验 |
| 核心问题 | 30 分钟 | 探索特定话题 |
| 收尾 | 5 分钟 | 最后的想法，后续步骤 |

**最佳实践：**

- 问开放式问题（"告诉我关于..."而不是"你喜欢..."）
- 多听少说（80/20 法则）
- 避免引导性问题
- 用"为什么"和"怎么"追问
- 记录准确的引用

### 研究综合

收集数据后，将其综合为可操作的洞察：

1. **转录** 访谈并整理笔记
2. **编码** 观察结果，标记主题
3. **聚类** 相关观察结果
4. **识别模式** 跨参与者的共同点
5. **创建洞察** 将发现与设计机会联系起来

## 用户画像

用户画像是代表你关键用户群的虚构角色。它们帮助团队保持同理心并做出以用户为中心的决策。

### 创建有效的用户画像

一个好的用户画像包括：

| 元素 | 描述 | 示例 |
|------|------|------|
| 名字 | 虚构但好记 | "经理 Sarah" |
| 照片 | 用于视觉参考的库存照片 | 与人口统计相关 |
| 人口统计 | 年龄、角色、地点、技术舒适度 | 34 岁，市场总监，城市 |
| 目标 | 他们想实现什么 | "减少花在报告上的时间" |
| 痛点 | 当前的挫败感 | "工具集成不好" |
| 行为 | 他们目前如何工作 | 早上第一件事看手机 |
| 场景 | 何时何地使用产品 | 通勤、办公室、家 |
| 引言 | 代表性陈述 | "我需要一目了然地看到数据" |

### 用户画像质量清单

| 标准 | 好 | 差 |
|------|----|----|
| 基于 | 真实研究数据 | 假设 |
| 具体性 | 具体细节 | 模糊的概括 |
| 可操作性 | 指导设计决策 | 放在文档里不用 |
| 数量 | 3-5 个主要画像 | 15+ 个画像 |
| 重点 | 行为和目标 | 仅人口统计 |

### 待办任务框架

用户画像的替代或补充：

| 组成部分 | 问题 | 示例 |
|----------|------|------|
| 任务 | 用户想完成什么？ | 跟踪项目进度 |
| 情境 | 何时/何地发生？ | 在每周团队会议期间 |
| 动机 | 为什么想要？ | 尽早发现阻碍 |
| 结果 | 成功是什么样子？ | 对状态有清晰的可见性 |

## 用户流程

用户流程图描绘了用户在产品中完成目标的路径。它们帮助设计师理解和优化用户旅程。

### 流程图元素

| 元素 | 符号 | 含义 |
|------|------|------|
| 开始/结束 | 圆角矩形 | 入口或出口点 |
| 过程 | 矩形 | 一个操作或屏幕 |
| 决策 | 菱形 | 是/否或多选点 |
| 箭头 | 带箭头的线 | 流程方向 |
| 文档 | 波浪矩形 | 文档或输出 |

### 流程类型

| 类型 | 范围 | 最佳用途 |
|------|------|----------|
| 任务流程 | 单一路径 | 简单、线性任务 |
| 用户流程 | 多路径 | 复杂决策 |
| 线框流程 | 屏幕 + 流程 | 可视化屏幕过渡 |
| 服务蓝图 | 端到端 | 完整服务体验 |

### 创建用户流程

1. **定义入口点**：用户如何到达？
2. **定义目标**：用户想完成什么？
3. **映射步骤**：用户采取什么行动？
4. **识别决策点**：路径在哪里分支？
5. **展示替代路径**：在每个决策点会发生什么？
6. **识别错误状态**：什么会出错以及如何处理？
7. **展示成功状态**：用户如何知道他们成功了？

### 示例：电商结账流程

```
浏览商品 → 加入购物车 → 查看购物车 → 结账
    |                                        |
    ↓                                        ↓
继续购物                            填写收货信息
                                           |
                                    [有效?]
                                     /      \
                                   是       否 → 显示错误
                                     |
                                 填写支付信息
                                     |
                                [有效?]
                                 /      \
                               是       否 → 显示错误
                                 |
                            订单确认
```

## 线框图

线框图是产品布局的低保真表示。它们专注于结构、内容层级和功能，而非视觉设计。

### 线框图保真度级别

| 级别 | 细节 | 工具 | 使用时机 |
|------|------|------|----------|
| 草图 | 手绘、粗糙 | 纸、白板 | 早期构思 |
| 低保真 | 基本形状，无样式 | Balsamiq、Whimsical | 概念验证 |
| 中保真 | 精确布局，灰度 | Figma、Sketch | 设计评审 |
| 高保真 | 详细，接近最终 | Figma、Sketch | 开发交付 |

### 线框图结构

每个线框图应清楚显示：

| 元素 | 目的 |
|------|------|
| 导航 | 用户如何在各部分之间移动 |
| 内容层级 | 页面上最重要的是什么 |
| 交互元素 | 按钮、链接、表单 |
| 信息展示 | 数据、文本、图片（占位符） |
| 状态 | 空、加载、错误、成功 |

### 线框图最佳实践

1. **使用真实内容**，而不是 Lorem Ipsum
2. **标注** 复杂交互并附注释
3. **展示多种状态**（空、填充、错误）
4. **考虑响应式布局**（手机、平板、桌面）
5. **保持简洁** -- 避免视觉设计干扰
6. **尽早获取反馈** -- 线框图修改成本低

## 原型设计

原型为线框图添加了交互性，让用户和利益相关者在产品构建之前就能体验它。

### 原型类型

| 类型 | 保真度 | 交互性 | 工作量 | 最佳用途 |
|------|--------|--------|--------|----------|
| 纸质 | 极低 | 物理 | 极低 | 快速概念测试 |
| 点击式 | 低-中 | 仅点击 | 低 | 流程验证 |
| 交互式 | 中-高 | 点击、悬停、输入 | 中 | 可用性测试 |
| 编码 | 高 | 完整 | 高 | 最终验证 |

### 原型工具对比

| 工具 | 学习曲线 | 协作 | 动画 | 价格 |
|------|----------|------|------|------|
| Figma | 低 | 优秀 | 基础 | 免费版 |
| ProtoPie | 中 | 好 | 高级 | 付费 |
| Principle | 中 | 无 | 时间轴 | 付费 |
| Framer | 高 | 好 | 基于代码 | 免费版 |
| Axure RP | 高 | 好 | 条件式 | 付费 |

### 什么需要做原型

不是所有东西都需要做原型。重点关注：

| 优先级 | 做原型的内容 | 原因 |
|--------|------------|------|
| 高 | 核心用户流程 | 最需要验证 |
| 高 | 复杂交互 | 从静态屏幕最难理解 |
| 中 | 新模式 | 用户不熟悉 |
| 低 | 简单页面 | 不需要交互也容易理解 |

## 可用性测试

可用性测试用真实用户验证你的设计。它揭示了对过于接近产品的设计师来说不可见的问题。

### 可用性测试方法

| 方法 | 参与者 | 时长 | 场景 |
|------|--------|------|------|
| 有人主持的面对面 | 5-8 人 | 30-60 分钟 | 实验室或办公室 |
| 有人主持的远程 | 5-8 人 | 30-60 分钟 | 视频通话 |
| 无人主持的远程 | 20-50 人 | 10-20 分钟 | 在线平台 |
| 游击测试 | 5-10 人 | 5-15 分钟 | 公共场所 |
| A/B 测试 | 1000+ | 数天/数周 | 生产环境 |

### 规划可用性测试

| 步骤 | 描述 |
|------|------|
| 定义目标 | 你想了解什么？ |
| 招募参与者 | 找到有代表性的用户 |
| 创建任务 | 让用户完成的真实场景 |
| 编写脚本 | 开场、任务、追问 |
| 试运行 | 运行一次测试来优化流程 |
| 执行测试 | 观察、记笔记、避免帮助 |
| 分析发现 | 识别模式和严重程度 |
| 报告结果 | 与团队分享洞察 |

### 任务编写指南

| 好的任务 | 差的任务 |
|---------|---------|
| "找一双 100 元以下的跑鞋" | "点击鞋子分类" |
| "你想退上周买的产品" | "去退货页面" |
| "设置账户接收邮件通知" | "点击设置，然后通知" |

### 严重程度评级

| 严重程度 | 描述 | 行动 |
|----------|------|------|
| 严重 | 用户无法完成任务 | 发布前修复 |
| 重大 | 用户困难但最终成功 | 下次迭代修复 |
| 轻微 | 用户注意到但可以绕过 | 方便时修复 |
| 外观 | 视觉或文案问题 | 有时间就修复 |

## 设计系统

设计系统是一组可复用组件的集合，由清晰的标准指导，可以组装来构建任意数量的应用。

### 设计系统组件

| 层级 | 描述 | 示例 |
|------|------|------|
| 设计 Tokens | 基础值 | 颜色、间距、字体 |
| 组件 | 可复用的 UI 元素 | 按钮、输入框、卡片 |
| 模式 | 常见解决方案 | 表单、导航、数据表 |
| 模板 | 页面布局 | 仪表板、设置、个人资料 |
| 指南 | 标准和规则 | 写作风格、无障碍 |

### 设计 Token 分类

| 类别 | 示例 |
|------|------|
| 颜色 | 主色、辅助色、语义色（错误、成功、警告） |
| 字体 | 字体族、大小、字重、行高 |
| 间距 | 4px、8px、16px、24px、32px 比例 |
| 圆角 | 小 (4px)、中 (8px)、大 (16px)、全 (9999px) |
| 阴影 | 低、中、高海拔 |
| 动效 | 持续时间（快、正常、慢）、缓动曲线 |

### 构建设计系统

| 阶段 | 活动 | 时长 |
|------|------|------|
| 审查 | 编目现有 UI 元素 | 1-2 周 |
| 定义 | 建立 tokens 和原则 | 2-4 周 |
| 构建 | 创建组件库 | 4-8 周 |
| 文档 | 编写使用指南 | 2-4 周 |
| 采用 | 集成到工作流程 | 持续 |
| 维护 | 更新和演进 | 持续 |

### 设计系统文档

每个组件应包括：

| 部分 | 内容 |
|------|------|
| 描述 | 组件是什么以及何时使用 |
| 结构 | 所有部分的标注图 |
| 变体 | 不同样式和尺寸 |
| 状态 | 默认、悬停、激活、禁用、错误 |
| 行为 | 如何响应交互 |
| 无障碍 | ARIA 角色、键盘导航 |
| 代码 | 实现示例 |
| 变更日志 | 版本历史 |

## 设计思维

设计思维是一种以人为中心的创新和问题解决方法。它整合了人的需求、技术的可能性和业务成功的要求。

### 五个阶段

| 阶段 | 目标 | 关键活动 |
|------|------|----------|
| 共情 | 理解用户 | 观察、参与、沉浸 |
| 定义 | 框定问题 | 综合分析、创建问题陈述 |
| 构思 | 生成解决方案 | 头脑风暴、草图、评估 |
| 原型 | 通过构建来思考 | 创建低成本的表示 |
| 测试 | 从用户那里学习 | 收集反馈、迭代 |

### 共情地图

共情地图捕捉用户说的、想的、做的和感受的：

| 象限 | 描述 | 研究来源 |
|------|------|----------|
| 说 | 直接引用 | 访谈 |
| 想 | 他们相信但可能不说的 | 访谈、观察 |
| 做 | 可观察的行为 | 情境调查 |
| 感受 | 情绪状态 | 访谈、观察 |

### "我们可以怎样"问题

将洞察转化为设计机会：

| 洞察 | "我们可以怎样"问题 |
|------|-------------------|
| 用户每天花 30 分钟写报告 | 我们可以怎样减少报告时间？ |
| 用户忘记更新状态 | 我们可以怎样让状态更新变得毫不费力？ |
| 新用户在引导流程中流失 | 我们可以怎样让引导更吸引人？ |

### 构思技巧

| 技巧 | 描述 | 最佳用途 |
|------|------|----------|
| 头脑风暴 | 不加评判地生成想法 | 想法的数量 |
| 疯狂 8 法 | 8 分钟内画 8 个草图 | 快速探索 |
| 思维导图 | 可视化想法分支 | 探索联系 |
| SCAMPER | 修改现有解决方案 | 渐进式创新 |
| 最差想法 | 故意想出坏主意 | 打破思维定式 |
| 故事板 | 用户体验的视觉叙事 | 理解上下文 |

## 衡量成功

好的产品设计是可衡量的。在发布前定义成功指标，以了解你的设计是否有效。

### 指标类别

| 类别 | 衡量内容 | 示例 |
|------|----------|------|
| 参与度 | 用户如何交互 | 会话数、页面停留时间、点击 |
| 任务成功 | 用户能否完成目标 | 完成率、错误率 |
| 效率 | 用户工作多快 | 任务时间、完成点击数 |
| 满意度 | 用户感受如何 | NPS、CSAT、SUS 分数 |
| 业务 | 对目标的影响 | 转化、留存、收入 |

### 关键 UX 指标

| 指标 | 公式 | 良好基准 |
|------|------|----------|
| 任务成功率 | (成功任务数 / 总任务数) x 100 | > 78% |
| 任务时间 | 完成平均时间 | 取决于上下文 |
| 错误率 | (错误数 / 总尝试数) x 100 | < 10% |
| 系统可用性量表 (SUS) | 10 题问卷，0-100 | > 68 |
| 净推荐值 (NPS) | 推荐者 - 贬损者 | > 0 |
| 客户满意度 (CSAT) | (满意回复 / 总数) x 100 | > 80% |

### 可用性目标框架

| 目标 | 指标 | 目标值 | 测量方式 |
|------|------|--------|----------|
| 有效性 | 任务完成率 | > 90% | 可用性测试 |
| 效率 | 平均任务时间 | < 2 分钟 | 数据分析 |
| 满意度 | SUS 分数 | > 80 | 问卷调查 |
| 可学习性 | 多次会话后的提升 | 快 20% | 重复测试 |
| 无障碍 | WCAG 合规 | AA 级 | 自动扫描 |

## 总结

产品设计是一个系统化的过程，结合同理心、创造力和数据来创建真正服务用户的产品。关键原则：

- **从研究开始**：永远不要仅基于假设设计
- **快速迭代**：尽早并经常用低保真原型测试
- **让用户参与**：他们的反馈是最有价值的输入
- **记录决策**：设计系统和模式确保一致性
- **衡量结果**：在发布前定义成功指标
- **协作**：设计是设计师、开发者、产品经理和利益相关者共同参与的团队运动
